### Integration Gaps

#### Gap 1: Photo URL public-path construction on private bucket (Severity: Critical)

**Affected requirements:** EVID-01, EVID-04

`photo-upload.controller.ts` line 46:
```typescript
const permanentUrl = `${supabaseUrl}/storage/v1/object/public/${BUCKET}/${path}`;
```

`005_photo_storage.sql` documents: `Public: FALSE (private bucket — access via signed URLs only)`

A `/storage/v1/object/public/...` URL on a private Supabase bucket returns an error. Photos stored in attendance records will render as broken images in `AttendanceRecordDetail` and `AttendanceHistoryTable` when the bucket is configured as documented.

The Supabase Storage bucket creation remains a pending TODO in STATE.md. This suggests the full photo display path may not have been live-tested against a real Supabase instance.

**Resolution options (pick one):**
1. **Set bucket to public** — simplest; path isolation (`{companyId}/{userId}/...`) provides tenant separation at the storage layer. Update documentation to match.
2. **Add signed read URL backend endpoint** — add `GET /attendance/photo-url?path=...` that calls `createSignedUrl()` for reading; frontend fetches signed URL before rendering `<img>`. More secure, matches original design intent.

#### Gap 2: No post-onboarding company settings UI (Severity: Medium — tech debt)

**Affected requirements:** ONBD-03 (partial), ONBD-04 (partial)

`GET /company/settings` backend endpoint exists but has no frontend consumer. Admin has no way to change timezone or IP mode after the onboarding wizard completes.

Requirements are technically satisfied (settings are configured during onboarding wizard). This is a UX gap, not a requirement violation, but should be addressed before v1 ships.
