This migration is intentionally minimal — only the timezone column is new schema. The orphan behavior is already correct from migration 004 (no cascade was specified, so PostgreSQL defaults to RESTRICT which blocks delete if records exist... wait — we need ON DELETE SET NULL or we need to remove the NOT NULL constraint, OR we use auth.admin.deleteUser which only deletes the auth user and we remove from public.users separately).

IMPORTANT CORRECTION: The attendance_records.user_id is `NOT NULL REFERENCES users(id)` — this means we CANNOT delete a user row if attendance records reference it, because PostgreSQL will block with a FK violation (RESTRICT is the default).

Architecture decision (per MEMORY.md): "Employee delete: removes auth account, keeps attendance records (orphaned)."

To make this work, we have two options:
1. Change attendance_records.user_id FK to ON DELETE SET NULL (and allow NULLs) — requires changing NOT NULL constraint
2. Keep user row in users table but remove from Supabase Auth only — soft delete approach: set is_active = false and delete auth user

Looking at the actual migration 004: need to read it to confirm the FK definition before deciding.

Since the requirement says "account disappears from active lists but every historical attendance record still shows their name" — this means we need to KEEP the users row (for name display) but remove their ability to log in (delete auth account + set is_active = false permanently). The "delete" is really a soft delete: the user disappears from active lists (filtered by is_active) but the row remains in users table so attendance records can still join and show full_name.

This is the correct interpretation. The migration only needs to add the timezone column. Update the SQL file content accordingly:

```sql
-- Migration 008: Employee Lifecycle + Per-User Timezone
-- Run via Supabase SQL editor before testing Phase 7 endpoints
-- Adds: users.timezone nullable column for per-employee timezone override

-- Add timezone override column to users table.
-- NULL = use company timezone (existing behavior preserved).

-- Sparse index — only rows where timezone is set
CREATE INDEX IF NOT EXISTS idx_users_timezone ON users(timezone) WHERE timezone IS NOT NULL;
```

NOTE: Employee "delete" is implemented as: (1) delete Supabase Auth user so they cannot log in, (2) mark is_active = false on public.users row so they disappear from active lists. The users row is RETAINED so attendance_records can join on user_id and still display the employee's full_name. No schema change needed for this behavior — it uses the existing is_active flag.
